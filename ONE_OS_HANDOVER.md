## Architecture Map

The system uses a **Vertical-Modular Architecture**, allowing shared core services to power diverse business types while maintaining isolation for domain-specific logic.

### 1. Core Services Layer (`/src/modules/core`, `auth`, `onboarding`)
- Handles identity, workspace switching, and the "First Run" experience.
- **Onboarding Flow**: `Identity` -> `Segment` -> `Type` -> `Workspace`.

### 2. The Vertical Layer (`/src/modules/commerce/*`)
- Each vertical (Restaurant, Pharmacy, Supermarket) is self-contained.
- **Manifest-Driven**: Each vertical exports a `VerticalManifest` defining its modules, paths, and metadata.
- **Vertical Routing**: Centrally managed in `src/views/routes.tsx` using layout wrappers.

### 3. Shared Commerce Components (`/src/modules/commerce/shared`)
- **Procurement Network**: A unified cross-vertical module located in `shared/procurement`.
- Verticals consume this via a wrapper (e.g., `SupermarketProcurementView`) passing their ID as a prop.

### 4. Shared UI Library (`/src/components/shared`)
- **ModuleShell**: Standardized wrapper for all business views.
- **PageHeader**: Context-aware headers with action support.
- **StatCard**: Reusable data visualization components.

## Database & API Integration: Procurement Network

### Schema Mapping (Types)
Defined in `src/types/procurement.ts`:
- **PurchaseOrder**: Root entity tracking `buyerId`, `supplierId`, `status`, and `total`.
- **PurchaseOrderItem**: Detailed line items with batch/expiry support (specific to Pharmacy).
- **SupplierProduct**: Catalog-level data for the procurement marketplace.

### API Integration Points
- **HTTP Client**: `src/api/http.ts` provides a global singleton for RESTful interactions.
- **Endpoint Pattern**: `/api/v1/[module]/...`
- **Authentication**: JWT-based via `Authorization: Bearer` header, managed by `auth.store`.
